## 1. JavaScript 基础

### Q: JavaScript 中的基本类型有哪些？各自如何存储？

**详解：**

* **7种原始类型（Primitive）**：`String`、`Number`、`Boolean`、`null`、`undefined`、`Symbol`（ES6）、`BigInt`（ES10）。
* **引用类型（Reference）**：`Object`（包含 `Array`、`Function`、`Date`、`RegExp` 等）。
* **存储机制**：
  * 原始类型：值大小固定，直接按值存放在**栈内存（Stack）**中，访问速度快。
  * 引用类型：值大小不固定，实体存放在**堆内存（Heap）**中，而栈内存中仅保存指向堆内存的**引用地址（指针）**。

### Q: 0.1 + 0.2 为什么不等于 0.3？如何解决？

**详解：**

* **原因**：JS 采用 IEEE 754 标准的**双精度浮点数（64位）**表示数字。十进制的 0.1 和 0.2 转换成二进制是无限循环小数。由于存储空间有限（尾数位只有 52 位），超出的部分会被截断，导致精度丢失。
* **解决方式**：
  1. 利用 `Number.EPSILON`（JS 最小精度值）进行判断：`Math.abs((0.1 + 0.2) - 0.3) < Number.EPSILON`。
  2. 转换为整数计算：`(0.1 * 10 + 0.2 * 10) / 10 === 0.3`。
  3. 使用第三方库：如 `Decimal.js` 或 `Big.js`。

### Q: typeof null 为什么是 'object'？

**详解：**
这是一个历史遗留 Bug。在 JS 最初的 32 位系统实现中，变量的类型信息由低位的标识位表示。`000` 开头代表对象（Object）。而 `null` 的所有位全是 0（表示空指针），因此它的低位也是 `000`，就被 JS 引擎错误地判断成了 `object`。由于修改此 Bug 会影响大量线上旧代码，官方决定不予修复。

### Q: 判断数组的方法有哪些？（高频）

**详解：**

1. **推荐**：`Array.isArray(arr)`（ES6 语法，最准确）。
2. **万能方法**：`Object.prototype.toString.call(arr) === '[object Array]'`。
3. **原型链判断**：`arr instanceof Array`（缺点：在多个 iframe 跨全局环境时，由于 Array 构造函数不同会导致判断失效）。
4. **构造函数**：`arr.constructor === Array`（缺点：`constructor` 可以被手动篡改）。

### Q: == 和 === 的区别？

**详解：**

* `===`（严格相等）：比较**类型和值**，两者必须完全相同，不进行类型转换。
* `==`（宽松相等）：如果类型不同，会发生**隐式类型转换**再比较。
  * **转换规则**：
    1. 字符串 vs 数字：字符串转数字（`'1' == 1` -> `true`）。
    2. 布尔 vs 其他：布尔转数字（`true == 1` -> `true`）。
    3. 对象 vs 原始类型：对象调用 `valueOf()` 或 `toString()` 转为原始值。
    4. `null == undefined` 为 `true`，但它们与其他任何值都不相等。

### Q: var、let、const 的区别？

| 特性       | `var`               | `let`           | `const`              |
|:-------- |:------------------- |:--------------- |:-------------------- |
| **作用域**  | 全局或函数作用域            | 块级作用域（`{}`内）    | 块级作用域                |
| **变量提升** | 提升且初始化为 `undefined` | 提升但不初始化（触发 TDZ） | 提升但不初始化（触发 TDZ）      |
| **重复声明** | 允许                  | 不允许             | 不允许                  |
| **初始赋值** | 不需要                 | 不需要             | 声明时必须初始化             |
| **重新赋值** | 允许                  | 允许              | 不允许（对于对象，不可改地址，可改属性） |

### Q: 什么是闭包？及其应用场景？

**详解：**

* **概念**：闭包是一个函数以及其捆绑的周边环境状态（词法环境）的引用的组合。简单说：**内层函数引用了外层函数的变量，就形成了闭包。**
* **核心特性**：函数执行完毕后，其作用域内的变量不会被垃圾回收，仍然保存在内存中供闭包函数访问。
* **应用场景**：
  1. **数据私有化**：模拟私有变量（如 Vue2 的 `data` 为什么是函数）。
  2. **柯里化（Currying）**：分批接收参数。
  3. **防抖节流**：内部维护 `timer` 变量。
  4. **模块化封装**：IIFE（立即执行函数）暴露接口。
* **缺点**：容易导致内存泄漏（变量无法及时释放）。

### Q: 箭头函数和普通函数的区别？

**详解：**

1. **没有自己的 `this`**：它的 `this` 是在定义时由**外层词法作用域**决定的，且永远无法通过 `call/apply/bind` 改变。
2. **不能作为构造函数**：不能使用 `new` 关键字，因为它没有 `[[Construct]]` 内部方法，也没有 `prototype` 属性。
3. **没有 `arguments` 对象**：只能通过剩余参数（`...args`）来获取所有实参。
4. **不能作为 Generator 函数**：不能使用 `yield` 关键字。

### Q: new 操作符执行时发生了什么？

**详解：**

1. 在内存中创建一个全新的空对象 `{}`。
2. 将这个新对象的内部原型 `__proto__` 指向构造函数的 `prototype` 属性。
3. 将构造函数内部的 `this` 绑定到这个新对象上，并执行构造函数内部代码（为新对象添加属性和方法）。
4. 判断构造函数的返回值：如果返回的是一个**对象/函数**，则返回该引用；如果返回的是**基本数据类型**或没有返回值，则返回第一步创建的新对象。

### Q: 原型和原型链是什么？

**详解：**

* **原型（Prototype）**：JS 中所有函数都有一个 `prototype` 属性（显式原型），指向一个对象（包含实例共享的方法）。所有对象都有一个 `__proto__` 属性（隐式原型），指向创建它的构造函数的 `prototype`。
* **原型链**：当我们访问对象的一个属性/方法时，如果对象自身没有，JS 引擎就会顺着 `__proto__` 去它的构造函数的 `prototype` 中找；如果还没有，就继续往上层对象的 `__proto__` 找，直到查找到 `Object.prototype.__proto__`（即 `null`）为止。这条查找的链条就是原型链。

## 2. JavaScript 进阶

### Q: JavaScript 的垃圾回收（GC）机制是什么？

**详解：**
V8 引擎采用**分代回收**策略，将内存分为**新生代**和**老生代**：

1. **新生代（存活时间短的对象）**：采用 **Scavenge 算法**。将内存分为 From 空间和 To 空间。分配对象时先在 From 分配，垃圾回收时，将 From 中存活的对象复制到 To 空间，然后清空 From，最后 From 和 To 角色对换。优点是快，缺点是空间利用率只有 50%。
2. **老生代（存活时间长、体积大的对象）**：采用 **标记-清除（Mark-Sweep）** 和 **标记-整理（Mark-Compact）** 算法。从根对象（全局 window 等）开始遍历，标记所有可达对象。然后清除未标记的对象。为了解决清除后产生的内存碎片，会通过“标记-整理”将存活对象向一端移动。
* **增量标记**：为了避免垃圾回收导致主线程长时间阻塞（Stop-The-World），V8 将标记过程分成一个个小段，与 JS 业务逻辑交替执行。

### Q: 四种常见的 JS 内存泄漏是什么？

**详解：**

1. **意外的全局变量**：未声明直接赋值的变量（如 `a = 1`）会挂载到 window 上，永远不回收。
2. **被遗忘的定时器或回调**：`setInterval` 未 `clearInterval`，导致其内部引用的变量无法释放。
3. **脱离 DOM 的引用**：DOM 节点被移除了，但在 JS 对象或数组中依然保留着对该 DOM 的引用。
4. **不规范的闭包**：闭包无意中引用了大量不再需要的数据。

### Q: 深度优先遍历（DFS）和广度优先遍历（BFS）？

**详解：**

* **DFS（深度优先）**：不撞南墙不回头。通常使用**递归**或**栈（Stack，后进先出）**实现。适用于查找目标处于较深层级的场景。
* **BFS（广度优先）**：层层递进。通常使用**队列（Queue，先进先出）**实现。适用于寻找最短路径。

## 3. ES6+

### Q: Map 和 Object 的区别？

**详解：**

1. **键的类型**：Object 的键只能是 String 或 Symbol；Map 的键可以是**任意类型**（包括对象、函数、NaN）。
2. **迭代与顺序**：Map 是**可迭代的（Iterable）**，可以直接用 `for...of` 遍历，且按插入顺序排序；Object 遍历需借助 `Object.keys()`。
3. **大小获取**：Map 直接通过 `map.size` 获取数量；Object 需要计算 `Object.keys(obj).length`。
4. **性能**：频繁增删键值对的场景，Map 的性能明显优于 Object。

### Q: 什么是 Proxy？和 Object.defineProperty 的对比？

**详解：**

* `Object.defineProperty`（Vue2 响应式原理）：
  * **缺点**：只能劫持对象的具体**属性**，如果对象新增/删除属性，它无法监听到（所以 Vue2 需要 `$set`）；无法原生监听数组的索引变化和 `length` 变化。
* `Proxy`（Vue3 响应式原理）：
  * **优点**：可以代理**整个对象**，拦截对对象的所有操作（包括属性增删、数组变异、函数调用等），提供了 13 种拦截陷阱（如 `get`, `set`, `has`, `deleteProperty`）。
  * **缺点**：兼容性稍差（不支持 IE），且无法被 Polyfill 完美降级。

## 4. 异步编程

### Q: 什么是 Promise？解决了什么问题？

**详解：**

* **本质**：Promise 是一个对象，代表了一个异步操作的最终完成（或失败）及其结果值。
* **三种状态**：`pending`（进行中）、`fulfilled`（已成功）、`rejected`（已失败）。**状态一旦改变，就不可逆转。**
* **解决的问题**：彻底打破了传统异步编程的**回调地狱（Callback Hell）**，通过链式调用（`.then()`）使得异步代码看起来像同步代码，并提供了集中式的错误处理（`.catch()`）。

### Q: 宏任务（MacroTask）和微任务（MicroTask）有哪些？执行顺序？

**详解：**

* **宏任务**：`script`（整体代码）、`setTimeout`、`setInterval`、I/O 操作、UI 渲染、`requestAnimationFrame`。
* **微任务**：`Promise.then/catch/finally`、`MutationObserver`、Node.js 中的 `process.nextTick`。
* **Event Loop（事件循环）执行顺序**：
  1. 执行栈首先执行**一个宏任务**（通常是整体 script 代码）。
  2. 同步代码执行完毕后，检查**微任务队列**。
  3. **清空所有微任务**（如果在执行微任务时又产生了新的微任务，会继续在当前回合清空）。
  4. 浏览器判断是否需要进行 UI 渲染。
  5. 进入下一轮，从宏任务队列取出一个宏任务执行。

## 5. HTML & CSS

### Q: 什么是 BFC？如何创建？有什么作用？

**详解：**

* **概念**：块级格式化上下文（Block Formatting Context）。它是一个独立的渲染区域，区域内部元素的布局不会影响到外部元素，反之亦然。
* **触发条件**（常见）：
  1. 根元素 `<html>`。
  2. 浮动元素：`float` 不为 `none`。
  3. 绝对定位：`position` 为 `absolute` 或 `fixed`。
  4. `overflow` 不为 `visible`（如 `hidden`, `auto`, `scroll`）。
  5. `display` 为 `inline-block`, `flex`, `grid` 等。
* **作用**：
  1. **清除浮动**：让父元素包含内部的浮动元素（防止高度塌陷）。
  2. **避免外边距重叠（Margin Collapse）**：同一个 BFC 下相邻块的 margin 会折叠，把元素放到不同 BFC 即可解决。
  3. **自适应两栏布局**：左边定宽浮动，右边触发 BFC，右边就不会被左边遮挡。

### Q: 水平垂直居中的方式？（手写 CSS）

**详解：**

1. **Flex 布局（最推荐）**：
   
   ```css
   .parent { display: flex; justify-content: center; align-items: center; }
   ```
2. **绝对定位 + transform（未知宽高）**：
   
   ```css
   .parent { position: relative; }
   .child { 
     position: absolute; top: 50%; left: 50%; 
     transform: translate(-50%, -50%); 
   }
   ```
3. **Grid 布局**：
   
   ```css
   .parent { display: grid; place-items: center; }
   ```

## 7. 浏览器原理

### Q: 从输入 URL 到页面呈现经历了什么？（核心高频）

**详解：**
**一、 网络阶段：**

1. **构建请求与缓存检查**：浏览器构建请求行，查找强缓存（`Cache-Control`）。如果有缓存且未过期，直接使用。
2. **DNS 解析**：浏览器缓存 -> 操作系统缓存 -> 路由器缓存 -> 递归/迭代查询 DNS 服务器，获取目标 IP。
3. **TCP 三次握手**：客户端与服务器建立可靠的 TCP 连接。如果是 HTTPS，还需要进行 TLS 握手。
4. **发送 HTTP 请求**：发送请求头和请求体。
5. **服务器响应**：服务器处理请求，返回 HTML 文档数据（状态码 200）。

**二、 渲染阶段（关键渲染路径）：**

1. **解析 HTML（Parse HTML）**：将 HTML 转化为 DOM 树。
2. **解析 CSS（Parse CSS）**：将 CSS 转化为 CSSOM 树。
3. **生成渲染树（Render Tree）**：合并 DOM 和 CSSOM，剔除不可见的元素（如 `display: none`）。
4. **布局计算（Layout/Reflow）**：计算每个节点在屏幕上的确切几何位置和大小。
5. **绘制（Paint）**：将渲染树转化为屏幕上的实际像素（调用 GUI 绘制指令）。
6. **合成（Composite）**：将不同的图层合并，显示在屏幕上。

### Q: 重排（Reflow）和重绘（Repaint）的区别与优化？

**详解：**

* **重排（回流）**：元素的**几何尺寸、位置**发生改变，浏览器需要重新计算布局（Layout）。代价极高。
* **重绘**：元素的**外观**发生改变（如 `color`, `background-color`），但不影响布局。浏览器只需重新绘制（Paint）。
* **结论**：**重排必定引发重绘，重绘不一定引发重排。**
* **优化方案**：
  1. 避免频繁操作样式，最好一次性修改 `class`。
  2. 使用 `transform` 替代 `top/left` 做动画（`transform` 只触发合成器图层，不触发重排和重绘，由 GPU 加速）。
  3. 使频繁变动的元素脱离文档流（`absolute/fixed`）。
  4. 避免频繁读取会导致浏览器清空渲染队列的属性（如 `offsetHeight`, `scrollTop`）。

## 8. 网络协议

### Q: GET 和 POST 的本质区别？

**详解：**
从 HTTP 规范的角度：

1. **语义**：GET 用于请求资源（幂等、安全）；POST 用于提交数据（不幂等、不安全）。
2. **传参位置**：GET 参数放在 URL 的 Query 参数中；POST 参数放在请求体（Body）中。
3. **缓存**：GET 请求默认会被浏览器缓存，也可以被收藏为书签；POST 不会。
4. **长度限制**：GET 请求的 URL 长度受限于浏览器和服务器（通常 2KB-8KB）；POST 无限制。
* *面试官扩展*：GET 会产生一个 TCP 数据包，POST 可能会产生两个（先发 Header，服务器回 100 Continue，再发 Body）。但这并非底层规范限制，而是特定网络环境和库（如 curl, 旧版浏览器）的实现。

### Q: 浏览器缓存机制（强缓存与协商缓存）？

**详解：**

1. **强缓存**（优先读取本地，不发网络请求，状态码 `200 from cache`）：
   * `Expires`（HTTP/1.0）：绝对过期时间，受限于客户端本地时钟，容易产生偏差。
   * `Cache-Control`（HTTP/1.1）：相对时间（如 `max-age=3600`）。优先级高于 Expires。
2. **协商缓存**（强缓存失效后，向服务器发请求验证，资源未改返回 `304 Not Modified`）：
   * `Last-Modified` / `If-Modified-Since`：基于文件最后修改时间。缺点是只能精确到秒，且有时文件重新生成但内容未变，也会导致缓存失效。
   * `ETag` / `If-None-Match`：基于文件内容的唯一哈希摘要。精确度最高，优先级高于 Last-Modified。

### Q: TCP 三次握手为什么不是两次？

**详解：**
主要为了**防止已失效的连接请求报文段突然又传送到了服务端，产生错误**。
设想一个场景：客户端发出的第一个 SYN 包在网络节点滞留了，于是客户端重发了第二个 SYN 包并完成了两次握手建立连接，通信完毕后关闭连接。此时，第一个滞留的 SYN 包到达服务端，服务端以为客户端又要建立连接，于是回复 SYN+ACK。如果只有两次握手，此时服务端就已经进入连接状态，等待客户端发送数据，而客户端其实根本没有发请求，这就造成了**服务端的资源白白浪费**。三次握手让客户端有机会发送 RST 拒绝连接。

## 10 & 11. Vue 与 React

### Q: Vue 的响应式原理？

**详解：**

* **Vue 2 (基于 Object.defineProperty)**：
  Vue 初始化时会遍历 data 对象，使用 `Object.defineProperty` 为所有属性添加 getter/setter。
  每个组件实例都有一个 `Watcher` 实例。当在模板中读取属性时触发 getter，进行**依赖收集**（将 Watcher 存入对应的 Dep 中）。当修改属性时触发 setter，Dep 会通知所有绑定的 Watcher 重新渲染视图。
* **Vue 3 (基于 Proxy)**：
  使用 ES6 的 `Proxy` 创建数据的代理对象，配合 `Reflect` 操作内置行为。
  优势：原生支持拦截对象属性的添加/删除、数组索引与长度修改，无需像 Vue 2 那样使用 `$set`；Proxy 是惰性递归，性能更好。

### Q: 虚拟 DOM (Virtual DOM) 为什么能提升性能？

**详解：**
直接操作真实 DOM 是非常昂贵的（每次操作都可能引发重排重绘）。
Virtual DOM 本质上是一个描述 DOM 结构的 **JavaScript 对象**。
当状态发生改变时：

1. 生成一棵新的 Virtual DOM 树。
2. 框架会使用 **Diff 算法**（Vue 的双端比较/快速 Diff，React 的 Fiber 协调机制）比较新旧两棵树。
3. 找出真正的差异点（Patch），然后**一次性、批量**更新到真实 DOM 上。
   *注意*：虚拟 DOM 并不是比原生原生 DOM API 操作更快，它的价值在于**提供了一种高效的声明式 UI 编程范式，并在多数复杂场景下保证了“下限不低”的性能表现**。

### Q: React 中 setState 是同步还是异步？

**详解：**

* **React 18 之前**：
  * 在 React 管辖的事件（如 `onClick` 等合成事件）和生命周期钩子中，它是**异步的（批处理）**，多次 setState 会合并为一次重新渲染。
  * 在非 React 管辖的事件（如 `setTimeout`、`setInterval`、原生 `addEventListener` 绑定的事件）中，它是**同步的**。
* **React 18 及以后**：
  引入了 `Automatic Batching`（自动批处理），**默认在所有情况下（包括 setTimeout 和原生事件）都是异步批处理的**。如果非要同步，需要使用 `flushSync` API。

## 12. 工程化（Webpack）

### Q: Loader 和 Plugin 的区别？

**详解：**

* **Loader（加载器）**：本质是函数。Webpack 默认只认识 JS 和 JSON。Loader 负责把其他文件（如 CSS、图片、Vue文件）转化成 Webpack 能识别的模块。例如 `babel-loader` (ES6转ES5)，`css-loader`。
* **Plugin（插件）**：本质是带有 `apply` 方法的类。它基于事件流架构，可以监听 Webpack 构建生命周期中的钩子函数（Hooks），在合适的时机执行任务。负责打包优化、资源管理等广泛任务。例如 `HtmlWebpackPlugin` (自动生成 HTML 并注入 JS)、`MiniCssExtractPlugin` (提取 CSS 为独立文件)。

### Q: 什么是 Tree Shaking？前提条件是什么？

**详解：**

* **概念**：字面意思是“摇树”，即把项目中没有用到的代码（Dead Code）在打包时直接删除，减小最终包的体积。
* **前提条件**：必须依赖于 **ES Module (ESM)** 规范（`import`/`export`）。因为 ESM 是**静态分析**的，Webpack 在编译阶段就能确定模块的依赖关系和哪些导出未被使用。传统的 CommonJS (`require`) 是动态运行时的，无法进行 Tree Shaking。

## 14. 安全

### Q: XSS 攻击是什么？如何防范？

**详解：**

* **概念**：跨站脚本攻击（Cross-Site Scripting）。黑客将恶意 JS 代码注入到网页中，当用户浏览时，恶意脚本获取用户的 Cookie 并在后台发送给黑客。
* **分类**：存储型（留言板存入数据库）、反射型（通过恶意 URL 诱导点击）、DOM 型。
* **防范**：
  1. **永远不信任用户输入**：对输入进行 HTML 实体转义（如把 `<` 转成 `&lt;`）。在 React/Vue 中默认开启了防注入。
  2. 开启 Cookie 的 **`HttpOnly`** 属性，使得 JS 无法通过 `document.cookie` 读取 Cookie。
  3. 配置 **CSP（内容安全策略）**：通过 HTTP 头 `Content-Security-Policy` 白名单限制只能加载特定域名的脚本。

### Q: CSRF 攻击是什么？如何防范？

**详解：**

* **概念**：跨站请求伪造（Cross-Site Request Forgery）。黑客诱导用户点击链接，利用用户在目标网站（如银行）**已经登录的 Cookie 会话凭证**，伪造用户的身份向目标网站发送恶意请求（如转账）。
* **防范**：
  1. **SameSite Cookie 属性**：设置为 `Strict` 或 `Lax`，禁止第三方网站携带 Cookie 发起请求（目前 Chrome 的默认行为）。
  2. **CSRF Token**：服务器渲染页面时生成一个随机 Token，每次提交表单或 Ajax 必须携带这个 Token，黑客网站无法拿到这个 Token。
  3. 验证 HTTP 请求头的 `Referer` 或 `Origin`，拒绝来自外部不受信任域名的请求。

## 15. 大厂面试真题解析

### Q: `['1', '2', '3'].map(parseInt)` 的执行结果？为什么？

**详解：**
结果是：`[1, NaN, NaN]`。

* `map` 函数会给回调传递三个参数：`(currentValue, index, array)`。
* `parseInt` 函数接收两个参数：`(string, radix)`。radix 是 2-36 的进制基数。
  结合起来，实际执行的是：
1. `parseInt('1', 0)`：radix 为 0 时默认为 10 进制，结果为 1。
2. `parseInt('2', 1)`：1 进制不存在，返回 NaN。
3. `parseInt('3', 2)`：2 进制中只能有 0 和 1，出现 3 是非法的，解析失败，返回 NaN。

### Q: 扫描二维码登录网页的底层原理是什么？（阿里高频）

**详解：**
这本质上是**PC端网页**与**手机端App**之间通过服务器建立关联的过程：

1. **生成二维码**：PC端打开网页，前端向服务器发请求，服务器生成一个唯一的 `UUID` 和带有此 UUID 的二维码，前端开始**长轮询（或 WebSocket）**监听该 UUID 的状态。
2. **手机扫码**：手机端（处于已登录态）扫描二维码，获取到了 `UUID`。
3. **状态变更（已扫码）**：手机端将 `UUID` 和自己的 `Token` 发给服务器。服务器将 UUID 状态改为“已扫码”。PC 网页端轮询到状态改变，UI 显示“扫描成功，请在手机上确认”。
4. **确认授权**：用户在手机上点击“确认登录”，手机发送确认请求给服务器。服务器验证后，为 PC 网页端生成正式的登录 `Token`，并改变 UUID 状态。
5. **PC端登录成功**：网页端轮询到最终态，获取到登录 Token，页面跳转登录成功。
