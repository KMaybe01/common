## 1. 模块化规范

### Q: CommonJS (CJS) 和 ES Module (ESM) 的区别是什么？（高频核心）
**详解：**
1. **加载方式**：
    * **CJS** 是**运行时加载**，同步操作。主要用于 Node.js 服务端。
    * **ESM** 是**编译时（静态）加载**，支持异步加载。可以在代码运行前就确定依赖关系。
2. **导出机制（值的拷贝 vs 值的引用）**：
    * **CJS** 导出的是值的**浅拷贝**。一旦输出一个值，模块内部的变化就**影响不到**这个值了。
    * **ESM** 导出的是值的**动态只读引用**。当模块内部的值改变时，外部导入的值也会**实时同步变化**。
3. **顶层 this**：CJS 的顶层 `this` 指向当前模块对象；ESM 的顶层 `this` 为 `undefined`。

### Q: 为什么 Tree Shaking 必须依赖 ES Module？
**详解：**
Tree Shaking（摇树优化）的核心是**死代码消除（Dead Code Elimination）**。
因为 CJS 是动态加载的（你可以把 `require` 写在 `if` 语句里），打包工具在运行前根本不知道模块导出了什么、被谁使用了。
而 ESM 是**静态的结构**，它的 `import` 和 `export` 必须写在模块顶层。Webpack 在**编译阶段（AST 语法树分析）**就能明确知道哪些变量被导入了、哪些根本没用到，从而把没用到的代码安全地“摇”掉。

## 2. 构建工具基石：Webpack

### Q: Webpack 的核心概念有哪些？
**详解：**
1. **Entry（入口）**：构建的起点，Webpack 从这里开始分析依赖图。
2. **Output（输出）**：告诉 Webpack 在哪里输出它所打包的文件，以及如何命名。
3. **Loader（加载器）**：Webpack 默认只理解 JS 和 JSON。Loader 可以将其他文件（如 CSS、Vue、图片）转换为有效的模块。
4. **Plugin（插件）**：用于执行范围更广的任务，贯穿 Webpack 的整个生命周期（如打包优化、资源管理、注入环境变量）。
5. **Mode（模式）**：区分 `development`（开启 HMR、保留 source map）和 `production`（开启 Tree Shaking、代码压缩）。

### Q: Loader 和 Plugin 的本质区别是什么？
**详解：**
* **Loader（转换器）**：
    * **本质**：是一个纯函数，接收源码内容作为参数，返回转换后的结果。
    * **作用时机**：在模块**解析加载（解析 AST 之前）**阶段执行。例如 `less-loader` 把 Less 转成 CSS，`babel-loader` 把 ES6 转成 ES5。
* **Plugin（扩展器）**：
    * **本质**：是一个具有 `apply` 方法的类。它基于 Webpack 的**Tapable 事件流架构**机制。
    * **作用时机**：贯穿 Webpack 的**整个生命周期**。通过监听特定阶段的钩子（Hooks），如 `compiler.hooks.emit`，来介入打包过程做更复杂的操作。例如 `HtmlWebpackPlugin` 生成首页，`MiniCssExtractPlugin` 抽离 CSS。

### Q: Webpack 的构建流程是怎样的？
**详解：**
这是一个串行的过程：
1. **初始化（Init）**：合并配置文件和 Shell 参数，实例化 `Compiler` 对象，并加载所有配置的 Plugins。
2. **编译启动（Run）**：调用 Compiler 的 `run` 方法开始构建。
3. **确定入口（Make）**：找到所有的 Entry。
4. **编译模块（Build Module）**：从入口出发，调用所有配置的 Loader 对模块进行翻译，然后将翻译后的内容转为 AST，找出该模块依赖的模块，**递归**进行本步骤直到所有依赖都经过本步骤的处理。
5. **完成编译（Seal）**：根据入口和模块之间的依赖关系，组装成一个个包含多个模块的 Chunk（代码块）。
6. **输出资源（Emit）**：将 Chunk 转换成文件，输出到文件系统（如 `dist` 目录）。

### Q: 如何优化 Webpack 的构建速度（打包太慢怎么调优）？
**详解：**
1. **缩小查找范围**：配置 Loader 的 `include` 和 `exclude`，比如排查 `node_modules`；配置 `resolve.alias` 和 `resolve.extensions` 减少路径解析。
2. **利用缓存**：
    * Webpack 5 之前使用 `cache-loader` 或 `HardSourceWebpackPlugin`。
    * **Webpack 5 内置了持久化缓存**：设置 `cache: { type: 'filesystem' }`，二次打包速度起飞。
3. **多进程打包**：使用 `thread-loader` 将耗时的 Babel 转换放在 worker 池中运行（注：进程启动也有开销，仅适合大型项目）。
4. **提前预编译**：配置 `externals` 忽略如 Vue, React 等大库，通过 CDN 引入。

### Q: 如何优化 Webpack 的打包体积？
**详解：**
1. **代码分割（Code Splitting）**：配置 `optimization.splitChunks`，将第三方库（`node_modules`）和业务公共代码单独抽离，避免单个 `app.js` 过大。
2. **Tree Shaking**：开启 `production` 模式自动去除无效代码。并在 `package.json` 中配置 `sideEffects: false` 告知 Webpack 没有副作用可以放心摇。
3. **动态导入（懒加载）**：路由组件使用 `import()` 语法按需加载。
4. **压缩代码**：使用 `TerserPlugin` 压缩 JS，`CssMinimizerPlugin` 压缩 CSS。
5. **图片优化**：小图片转 Base64（Webpack5 使用 `type: 'asset/inline'`），大图片使用 `image-webpack-loader` 压缩。

### Q: Webpack 热更新（HMR）的原理是什么？
**详解：**
1. Webpack-dev-server 在**浏览器**和**本地 Node 服务**之间建立了一个 **WebSocket 长连接**。
2. 当本地代码发生修改并被 Webpack 重新编译后，服务端会向浏览器推送一条消息（包含新的 hash 值）。
3. 浏览器拿到 hash 后，通过 HTTP 请求向服务端获取一个 `manifest.json` 文件（知道哪些模块变了）和一个 `.js` 补丁文件（增量更新代码）。
4. 浏览器端通过 HMR Runtime 机制，使用新代码替换老代码（执行 `module.hot.accept` 回调），从而实现无刷新更新。

## 3. 下一代构建工具：Vite 与 Rollup

### Q: 为什么 Vite 在开发环境下比 Webpack 快那么多？（高频核心）
**详解：**
* **Webpack（Bundle 机制）**：冷启动时，必须抓取整个项目的路由和所有模块，递归构建出一整张依赖图，**全部打包编译后**才启动 Dev Server。项目越大，启动越慢。
* **Vite（No-Bundle 机制）**：
    1. Vite 利用了现代浏览器**原生支持 ES Module (`<script type="module">`)** 的特性。它把启动服务器和编译分离，冷启动瞬间完成。
    2. 当浏览器发起对某个具体文件的请求时，Vite 服务器才会**按需拦截请求**并进行转换后返回。只编译当前屏幕需要的代码。
    3. **预构建加速**：Vite 使用 Go 语言编写的 **Esbuild** 对第三方依赖（如 `lodash`、`react`）提前进行预构建并转为 ESM 格式，Esbuild 的速度比 JS 编写的打包器快 10-100 倍。

### Q: Vite 生产环境为什么要用 Rollup 打包，而不是直接用 Esbuild？
**详解：**
开发环境用 Esbuild 是为了**极速**；但生产环境更看重打包体积、代码分割的最佳实践和生态兼容性。
虽然 Esbuild 极快，但它在生产环境的**代码分割（Code Splitting）、CSS 处理、以及旧版浏览器的降级兼容**方面目前还不成熟。
Rollup 是一个极其优秀的 ESM 打包器，其产物更干净纯粹，且拥有一套非常完善的插件生态。Vite 在生产环境基于 Rollup，能提供更稳定、优化更好的线上代码。

## 4. 包管理器：npm / yarn / pnpm

### Q: npm/yarn 发展过程中的嵌套问题与幽灵依赖？
**详解：**
* **早期 npm v2（嵌套树结构）**：如果包 A 和包 B 都依赖了包 C，包 C 会被下载两次并分别存放在 A 和 B 的 `node_modules` 里。导致**路径过长**和**磁盘占用极大**。
* **npm v3 / yarn（扁平化结构）**：为了解决嵌套问题，所有的依赖被“铺平”提升到了顶层的 `node_modules`。
* **幽灵依赖（Phantom Dependencies）的产生**：由于扁平化，哪怕你在 `package.json` 中根本没有声明安装包 C，但因为它是包 A 的底层依赖被提升到了顶层，你在代码里竟然能直接 `require('C')` 并运行成功！如果某天包 A 升级移除了包 C，你的项目就会突然报错崩溃。这就叫“幽灵依赖”。

### Q: 为什么现在推荐使用 pnpm？它是怎么解决幽灵依赖的？
**详解：**
pnpm 结合了**硬链接（Hard Link）**和**符号链接（软链接 Symlink）**的思想：
1. **极速与省空间**：pnpm 全局维护一个存储中心（Store）。当你安装依赖时，如果在全局中心已经有了，pnpm 会通过“硬链接”直接指向它，不会在本地硬盘复制第二份实体。
2. **非扁平化、解决幽灵依赖**：pnpm 的项目的 `node_modules` 里只有你真正在 `package.json` 里声明的包。这些包其实是**软链接**，指向嵌套极深的 `.pnpm/` 虚拟存储目录。这就使得你无法引用 `package.json` 中未声明的依赖，彻底消灭了幽灵依赖问题，项目结构极其严谨。

### Q: package-lock.json 的作用是什么？
**详解：**
由于 `package.json` 中的版本号通常带有 `^` (兼容小版本) 或 `~` (兼容补丁版本) 前缀。导致今天安装和三个月后安装，实际拉取的依赖子版本可能不同，容易引发诡异的 Bug。
`package-lock.json`（或 `yarn.lock` / `pnpm-lock.yaml`）锁定了当前成功安装的所有依赖的**精确版本号、下载地址及其完整树状结构**。保证团队中任何人在任何时间 `npm install` 得到的依赖环境完全一致。

## 5. 代码规范与自动化

### Q: ESLint 和 Prettier 的分工是什么？经常有冲突怎么解决？
**详解：**
* **ESLint**：主要关注代码的**逻辑质量**（Code Quality）。比如禁止使用未定义的变量、强制使用 `const`、避免不必要的闭包等，顺带管一点格式。
* **Prettier**：只关注代码的**格式美观**（Formatting）。比如缩进是 2 个空格还是 4 个、句尾加不加分号、单引号还是双引号。
* **冲突原因与解决**：ESLint 也有自己的格式化规则，当 ESLint 认为应该用单引号，而 Prettier 格式化为双引号时，就会报错。
  **解决方案**：安装 `eslint-config-prettier` 插件，将它放在 ESLint 配置的 `extends` 数组的最后一位，它的作用就是**直接禁用掉所有和 Prettier 起冲突的 ESLint 格式化规则**，让 ESLint 专心查 Bug，让 Prettier 专心做美化。

### Q: 什么是 Husky 和 lint-staged？
**详解：**
仅仅有 ESLint 配置不够，因为开发者可能会忘记运行格式化就直接 `git commit`。这就需要工程化手段强制约束。
* **Husky**：是 Git Hooks 的工具，能够拦截 `pre-commit`（提交前）、`commit-msg`（写提交信息时）等 Git 生命周期的动作。
* **lint-staged**：如果我们在 `pre-commit` 时对整个项目跑一遍 ESLint，太慢了。`lint-staged` 的作用是**只对目前在 Git 暂存区（staged）被修改的文件**进行 Lint 检查和格式化。极大地提高了提交效率。
* **标准工作流**：执行 `git commit` -> 触发 Husky `pre-commit` 钩子 -> 执行 `lint-staged` -> 只检查本次改动的文件 -> 检查通过，成功 Commit；检查失败，驳回提交。

### Q: 你了解常规的 Git Commit 规范吗？
**详解：**
目前主流使用的是 **Conventional Commits** 规范（由 Angular 团队提出）。
提交信息格式通常为：`<type>(<scope>): <subject>`。
常见的 type 有：
* `feat`: 新增功能（feature）
* `fix`: 修复 bug
* `docs`: 仅修改文档（如 README）
* `style`: 格式修改（不影响代码逻辑，如空格、分号）
* `refactor`: 重构（既不是新增功能，也不是修改 bug）
* `perf`: 性能优化
* `chore`: 构建过程或辅助工具的变动（如更新 npm 包）
  *可以通过 `Commitizen` 工具进行交互式规范提交。*

## 6. Babel 编译原理

### Q: Babel 是如何把 ES6 代码转成 ES5 的？（Babel 的工作流程）
**详解：**
Babel 的本质是一个**源码到源码的编译器（Transpiler）**。主要分为三步：
1. **解析（Parse）**：读取源代码文本，通过词法分析和语法分析，将其转化为**抽象语法树（AST）**。（工具：`@babel/parser`）
2. **转换（Transform）**：深度优先遍历 AST，执行开发者配置的各个插件（Plugin）。插件会找到特定的节点，对其进行增、删、改，生成一棵新的 AST。（工具：`@babel/traverse`）
3. **生成（Generate）**：将转换后的新 AST 重新翻译回 JS 字符串代码，同时生成 Source Map 文件。（工具：`@babel/generator`）

### Q: Babel 中的 Plugin 和 Preset 有什么区别？执行顺序是怎样的？
**详解：**
* **Plugin（插件）**：Babel 的核心功能就是靠一个个插件实现的。每一个插件只干一件极其具体的事（比如 `@babel/plugin-transform-arrow-functions` 只负责转译箭头函数）。
* **Preset（预设）**：因为 ES6+ 的新语法太多了，一个一个配插件会疯掉。Preset 是一组**预先配置好的插件大礼包**。例如 `@babel/preset-env` 包含了所有最新的 JavaScript 标准插件。
* **执行顺序（非常反直觉，必考）**：
    1. Plugin 会比 Preset **先**执行。
    2. Plugin 按照数组配置的顺序从**左到右（从前到后）**执行。
    3. Preset 按照数组配置的顺序从**右到左（从后到前）**执行。（因为早期作者希望兼容 C++ 的链式洋葱圈模型思想）。

### Q: Polyfill（垫片）的作用是什么？和普通的语法转译有什么区别？
**详解：**
Babel 默认只转换**新语法（Syntax）**（如 `let`, `const`, 箭头函数, class）。
但它**不会转换新的全局变量和 API（内置对象/实例方法）**（如 `Promise`, `Map`, `Set`, `Array.prototype.includes`）。
因为这些 API 在旧浏览器中完全不存在，必须通过注入兼容代码来实现，这就叫做 **Polyfill（垫片）**。
通常在工程化中，我们会配置 `@babel/preset-env` 并设置 `useBuiltIns: 'usage'`，配合 `core-js`，实现**按需引入 Polyfill**，以最小化打包体积。

## 7. CI/CD 与前端部署

### Q: 什么是 CI/CD？在前端工程中通常怎么配置？
**详解：**
* **CI（Continuous Integration，持续集成）**：当代码被 push 到仓库时，自动化触发脚本进行构建、Lint 检查、单元测试。尽早发现集成错误。
* **CD（Continuous Delivery/Deployment，持续交付/部署）**：在 CI 通过的基础上，将构建好的产物（静态资源）自动上传、部署到测试服务器或生产环境服务器（CDN 等）。
* **常见工具**：GitLab CI、GitHub Actions、Jenkins。
* **常规前端流水线（Pipeline）**：
    1. `Install` 阶段：`npm ci`（比 `npm i` 更严格的安装）。
    2. `Lint` 阶段：`npm run lint` 检查代码规范。
    3. `Build` 阶段：`npm run build` 打包项目。
    4. `Deploy` 阶段：通过 SSH/FTP/Docker 将 dist 目录推送到 Nginx 服务器或 OSS（对象存储）+ CDN。

### Q: 前端工程化部署中，静态资源的缓存策略是怎样的？（为什么打包出来的文件名带有一串 hash？）
**详解：**
这是前端性能优化的终极工程化方案，核心原则是：**"非阻塞加载与强缓存最大化"**。
1. **HTML 文件的策略**：**千万不能缓存（强缓存）**！通常设置为 `Cache-Control: no-cache` 或协商缓存。因为 HTML 是入口，如果它被缓存了，用户将永远拿不到最新的静态资源链接。
2. **JS/CSS/图片文件的策略**：设置**最长的强缓存**时间（如 `Cache-Control: max-age=31536000` 即一年）。
3. **如何更新？**：Webpack/Vite 打包时，会根据文件内容生成一个摘要（Hash/ChunkHash/ContentHash）拼接到文件名中（如 `app.[contenthash].js`）。
    * 文件内容不改，hash 不变，浏览器完美命中本地长缓存。
    * **文件内容一旦修改，重新打包后生成的新 hash 文件名一定不同**。部署时，新的 HTML 引入的是新的文件名，用户访问时就会下载最新的资源，而完全不需要手动清空浏览器缓存。

### Q: 单页应用（SPA）部署到 Nginx 服务器后，刷新页面出现 404，为什么？怎么解决？
**详解：**
* **原因**：SPA 通常使用 History 模式的前端路由。比如在浏览器访问 `https://abc.com/user/profile`，这个路径仅仅在前端 JS 解析时有意义。但是当你在当前页面**按下刷新（F5）**时，浏览器会拿着 `/user/profile` 这个完整路径去向 Nginx 服务器请求文件。而服务器的静态目录里只有一个根目录的 `index.html`，并没有 `/user/profile` 这个真实的物理文件，所以报 404 Not Found。
* **解决**：利用工程化思想修改服务器端配置。在 Nginx 中配置 **try_files**。
  ```nginx
  location / {
      root /usr/share/nginx/html;
      index index.html index.htm;
      # 核心原理：如果按路径找不到具体的文件或目录，就兜底返回 index.html。
      # 然后交给 index.html 里面的前端 vue-router/react-router 接管路由解析。
      try_files $uri $uri/ /index.html;
  }
  ```
